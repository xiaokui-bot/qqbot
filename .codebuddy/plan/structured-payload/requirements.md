# 需求文档：QQBot 结构化消息载荷优化

## 引言

本文档定义了 QQBot 与 OpenClaw 之间通信的优化方案。通过引入结构化消息载荷格式，解决当前系统中存在的以下问题：

1. **定时提醒场景**：纯文本消息导致上下文信息丢失
2. **发送图片场景**：依赖 Markdown 语法存在解析歧义和误触发风险
3. **发送音频场景**：需要统一的媒体处理机制
4. **接收图片场景**：需要将图片信息转换为 AI 易于理解的格式

### 最终方案概览

| 操作类型 | AI 输出格式 |
|---------|-------------|
| 定时提醒 | `QQBOT_PAYLOAD: { "type": "cron_reminder", ... }` |
| 发送图片 | `QQBOT_PAYLOAD: { "type": "media", "mediaType": "image", ... }` |
| 发送音频 | `QQBOT_PAYLOAD: { "type": "media", "mediaType": "audio", ... }` |

**代码层统一处理流程**：AI 回复 → 检测 `QQBOT_PAYLOAD:` → 解析 JSON → 根据 `type` 分发处理

### 核心设计原则

| 通信方向 | 处理者 | 最佳格式 | 原因 |
|---------|--------|----------|------|
| AI → 代码层（发送） | 代码解析 | 结构化 JSON | 代码需要精确解析 |
| 代码层 → AI（接收） | AI 理解 | 自然语言描述 | AI 擅长理解自然语言 |

### 重要技术决策

**Base64 编码由代码层负责，而非 AI**：大模型对 Base64 编码/解码不可靠（本质是 token 预测，非精确计算），因此 AI 只输出 JSON 文本，由代码层完成 Base64 编码。

## 需求

### 需求 1：统一结构化消息载荷格式（AI → 代码层）

**用户故事：** 作为一名开发者，我希望有一个统一的结构化消息格式，以便代码层能够精确解析 AI 输出的各类消息指令。

#### 验收标准

1. WHEN AI 需要发送结构化指令 THEN 系统 SHALL 使用 `QQBOT_PAYLOAD:` 前缀标识
2. WHEN 代码层检测到 `QQBOT_PAYLOAD:` 前缀 THEN 系统 SHALL 提取后续 JSON 内容并解析
3. WHEN 需要调用 OpenClaw 命令 THEN 代码层 SHALL 负责将 JSON 编码为 Base64
4. IF 消息不包含 `QQBOT_PAYLOAD:` 前缀 THEN 系统 SHALL 将其作为普通文本消息处理
5. WHEN 解析 JSON 获取 `type` 字段 THEN 系统 SHALL 根据 `type` 值分发到对应的处理器

#### 载荷格式定义

```
QQBOT_PAYLOAD:
{json_content}
```

#### 支持的载荷类型

| type 值 | 用途 | 处理方式 |
|---------|------|----------|
| `cron_reminder` | 定时提醒 | 编码 Base64 后调用 openclaw cron |
| `media` | 媒体消息（图片/音频等） | 解析后调用对应的发送 API |

**AI 输出示例**（AI 只需输出 JSON，无需 Base64）：
```
QQBOT_PAYLOAD:
{
  "type": "cron_reminder",
  "content": "💧 喝水时间到！",
  "targetType": "c2c",
  "targetAddress": "xxx_user_openid"
}
```

### 需求 2：定时提醒消息载荷

**用户故事：** 作为一名用户，我希望设置的定时提醒能够携带完整的上下文信息，以便收到提醒时能够准确理解提醒内容。

#### 验收标准

1. WHEN AI 设置定时提醒 THEN 系统 SHALL 输出包含以下字段的 JSON 载荷：
   - `type`: 固定为 `"cron_reminder"`
   - `content`: 实际提醒文本
   - `targetType`: 目标类型（`"c2c"` 或 `"group"`）
   - `targetAddress`: 目标地址（user_openid 或 group_openid）
   - `originalMessageId`: 原始消息 ID（可选）

2. WHEN 代码层处理定时提醒载荷 THEN 系统 SHALL：
   - 将 JSON 编码为 Base64
   - 添加 `QQBOT_CRON:` 前缀
   - 调用 `openclaw cron add --message "QQBOT_CRON:{base64}"`

3. WHEN OpenClaw 触发定时任务 THEN 系统 SHALL：
   - 检测 `QQBOT_CRON:` 前缀
   - 解码 Base64 获取 JSON
   - 根据 `targetType` 和 `targetAddress` 发送 `content` 内容

#### 实际可获得字段说明

| 消息类型 | 可获得字段 |
|---------|-----------|
| C2C 消息 | `author.user_openid`, `content`, `id`, `timestamp` |
| 群消息 | `author.member_openid`, `group_openid`, `content`, `id`, `timestamp` |

**注意**：用户昵称等字段无法获取，载荷中不应包含无法获取的字段。

#### 示例载荷

```json
{
  "type": "cron_reminder",
  "content": "💧 喝水时间到！记得补充水分哦~",
  "targetType": "c2c",
  "targetAddress": "ABC123_user_openid",
  "originalMessageId": "msg_xxx_123"
}
```

### 需求 3：媒体消息载荷（统一媒体处理）

**用户故事：** 作为一名用户，我希望 AI 能够可靠地发送图片、音频等媒体文件，以便获得更丰富的交互体验。

#### 验收标准

1. WHEN AI 需要发送媒体文件 THEN 系统 SHALL 输出包含以下字段的 JSON 载荷：
   - `type`: 固定为 `"media"`
   - `mediaType`: 媒体类型（`"image"` | `"audio"` | `"video"` 等）
   - `source`: 来源类型（`"url"` 或 `"file"`）
   - `path`: 媒体路径或 URL
   - `caption`: 媒体描述（可选）

2. WHEN 代码层处理媒体载荷 THEN 系统 SHALL：
   - 解析 JSON 获取媒体信息
   - 根据 `mediaType` 分发到对应处理逻辑
   - 根据 `source` 类型处理媒体（URL 直接发送，file 需要上传）

3. IF 媒体发送失败 THEN 系统 SHALL 记录错误日志并返回友好的错误提示

#### 支持的 mediaType

| mediaType | 说明 | 处理方式 |
|-----------|------|----------|
| `image` | 图片 | 调用图片发送 API |
| `audio` | 音频 | 调用音频发送 API |
| `video` | 视频（预留） | 调用视频发送 API |

#### 示例载荷

**发送图片**：
```json
{
  "type": "media",
  "mediaType": "image",
  "source": "url",
  "path": "https://example.com/image.jpg",
  "caption": "这是一张示例图片"
}
```

**发送音频**：
```json
{
  "type": "media",
  "mediaType": "audio",
  "source": "file",
  "path": "/path/to/audio.mp3",
  "caption": "这是一段语音消息"
}
```

### 需求 4：接收图片处理（代码层 → AI）

**用户故事：** 作为一名用户，我希望发送图片给 AI 时，AI 能够准确理解图片的上下文信息。

#### 验收标准

1. WHEN 用户发送图片 THEN 系统 SHALL 将图片信息转换为自然语言描述
2. WHEN 构建自然语言描述 THEN 系统 SHALL 包含以下信息：
   - 图片 URL 或路径
   - 图片格式（如有）
   - 图片尺寸（如有）
   - 消息 ID
   - 时间戳

3. WHEN 将描述传递给 AI THEN 系统 SHALL 使用清晰的自然语言格式，而非 JSON

#### 自然语言描述格式

```
用户发送了一张图片：
- 图片地址：{url}
- 图片格式：{format}
- 消息ID：{messageId}
- 发送时间：{timestamp}

请根据图片内容进行回复。
```

**设计原因**：大模型擅长理解自然语言，而非结构化 JSON。将图片信息转换为自然语言描述，能让 AI 更直观地理解上下文。

---

### 需求 5：AI 提示词更新

**用户故事：** 作为一名开发者，我希望 AI 能够正确输出结构化载荷格式，以便系统能够正确解析和处理。

#### 验收标准

1. WHEN 配置 AI 提示词 THEN 系统 SHALL 包含以下指导：
   - 定时提醒使用 `QQBOT_PAYLOAD:` + JSON 格式，type 为 `cron_reminder`
   - 发送图片使用 `QQBOT_PAYLOAD:` + JSON 格式，type 为 `media`，mediaType 为 `image`
   - 发送音频使用 `QQBOT_PAYLOAD:` + JSON 格式，type 为 `media`，mediaType 为 `audio`
   - 明确说明 AI 只需输出 JSON，Base64 编码由代码处理

2. WHEN AI 输出结构化消息 THEN 系统 SHALL 确保 JSON 格式正确且字段完整

---

### 需求 6：移除旧格式支持

**用户故事：** 作为一名开发者，我希望系统完全使用新的结构化格式，以便简化代码逻辑并避免歧义。

#### 验收标准

1. WHEN 处理 AI 输出 THEN 系统 SHALL 不再解析旧 Markdown 图片语法 `![](path)`
2. WHEN 处理消息 THEN 系统 SHALL 不再处理内部标签如 `[[reply_to:xxx]]`
3. IF 消息不包含 `QQBOT_PAYLOAD:` 前缀 THEN 系统 SHALL 直接作为普通文本处理

#### 需要删除的旧代码

- `collectImageUrl()` 或类似的 Markdown 图片解析函数
- `![](...)` 正则匹配逻辑
- `[[reply_to:xxx]]` 等内部标签处理逻辑

## 成功标准

1. ✅ 定时提醒能够携带完整上下文信息（消息ID、目标地址等）
2. ✅ 图片发送通过结构化载荷实现，无歧义
3. ✅ 音频发送通过结构化载荷实现，与图片使用相同机制
4. ✅ 接收图片时，AI 能够通过自然语言描述理解图片上下文
5. ✅ 移除所有旧格式解析逻辑
6. ✅ Base64 编码完全由代码层处理，AI 只输出 JSON
7. ✅ 载荷中只包含实际可获得的字段，不包含无法获取的字段
8. ✅ 代码层实现统一的 `type` 分发机制，易于扩展新类型

---

## 技术约束

1. **大模型限制**：AI 不擅长 Base64 编码/解码，所有编码工作由代码层完成
2. **字段限制**：C2C 消息只能获取 `user_openid`，群消息只能获取 `member_openid` 和 `group_openid`，用户昵称无法获取
3. **格式统一**：所有结构化消息统一使用 `QQBOT_PAYLOAD:` 前缀
4. **扩展性**：通过 `type` 字段实现类型分发，新增类型只需添加对应处理器
